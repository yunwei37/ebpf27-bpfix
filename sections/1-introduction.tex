\section{Introduction}
\label{sec:introduction}

eBPF lets developers run custom programs inside the Linux kernel for networking, security, and observability~\cite{gbadamosi2024ebpf, cilium}.
% eBPF 让开发者在内核里、网络/安全/可观测性的关键路径上运行自定义程序。
To load a program, a developer writes it in C or Rust, compiles it to eBPF bytecode, and submits it to a verifier. The verifier proves the program safe by tracking abstract values along every execution path~\cite{gershuni2019simple, linuxkernel-bpf-verifier}.
% 开发者用 C 或 Rust 写程序、编译成 BPF 字节码、提交给静态 verifier，verifier 沿每条执行路径跟踪抽象值来证明程序安全。

% 段2: 问题 + 例子
However, the verifier reports where it stopped, not where the proof was lost, making repair challenging.
% 然而，verifier 报告的是它停下的位置，而非 proof 丢失的位置，这使得修复困难。
Figure~\ref{fig:overview} illustrates this diagnostic gap: a packet pointer that the program bounds-checks is rejected at a later load with \texttt{R5 invalid mem access `scalar'}, on a line the developer has no reason to suspect.
% 图~\ref{fig:overview} 展示了这个 diagnostic gap：程序已经 bounds-check 过的 packet pointer 仍在后续 load 处被拒，而出错的行开发者根本想不到有问题。
Without a way to locate where the proof was lost, developers must rely on repeated trial and error~\cite{deokar2024empirical}.
% 无法定位 proof 在哪丢失，开发者只能依赖反复试错。

% 段3: Empirical study
To quantify this gap, we conduct an empirical study of 235 rejections reproduced from Stack Overflow, GitHub issues, fix commits, and kernel selftests.
% 为量化这一 gap，我们对来自 Stack Overflow、GitHub issue、修复 commit 和内核 selftest 的 235 个复现 rejection 进行实证研究。
We find that 191 are program bugs spanning 12 root causes, while the other 44 reject correct source due to compiler optimizations that obscure pointer types, environment misconfiguration, or verifier limitations.
% 我们发现 191 个是程序 bug、涵盖 12 类 root cause，另外 44 个拒绝的是正确源码，原因是编译器优化隐藏了指针类型、环境配置错误或 verifier 的限制。
Of the 12 root causes, 10 are eBPF-specific (e.g., packet bounds, dynptr lifetime), so repair requires domain knowledge beyond general programming.
% 12 类 root cause 中有 10 类是 eBPF 特有的（如 packet bounds、dynptr lifetime），因此修复需要超越通用编程的领域知识。
The terminal error provides little help: 47\% of rejections return \texttt{EINVAL}, and one error string maps to as many as nine distinct root causes.
% terminal error 帮助甚少：47\% 的 rejection 返回 \texttt{EINVAL}，一个 error 字符串最多对应 9 种不同的 root cause。

% 段4: 现有工具不足
Existing tools do not close this gap.
% 现有工具无法弥合这个 gap。
PrettyVerifier~\cite{rizza2025design} and the BPF Verifier Visualizer~\cite{bpfvv} help developers read the error with regular expressions, but neither locates where the proof was lost.
% PrettyVerifier 和 BPF Verifier Visualizer 帮助开发者阅读 error，但都不能定位 proof 在哪丢失。
Localizers in other domains like Rust walk a structure the checker builds, such as an AST or typing constraints~\cite{pavlinovic2014finding, zhang2014toward}, but the eBPF verifier exposes no such artifact.
% 其他领域的定位器走 checker 构建的结构（如 AST 或 typing constraints），而 eBPF verifier 不暴露这类制品。

% 段5: \sys
We introduce \sys, which reconstructs where the required proof was established and where it was lost from the verifier log and print a Rust-like diagnostic (Figure~\ref{fig:overview}).
% 我们提出 \sys，它从 verifier log 重建所需 proof 在哪建立、在哪丢失，并打印类似 Rust 的诊断信息（图~\ref{fig:overview}）。

% 段6: LLM benchmark 结果
To evaluate whether this localization helps repair, we construct \bench, a benchmark of 75 repair tasks with independent tests.
% 为评估这种定位是否有助于修复，我们构建 \bench：一个带独立测试的 75 修复任务 benchmark。
Current LLMs achieve 0--37\% one-shot repair with the raw verifier log; replacing the log with the \sys localization improves repair by 11--21pp across three models ranging from 3B to 27B parameters.
% 当前 LLM 用 raw verifier log 的 one-shot 修复为 0--37\%；换成 \sys 的定位后，三个规模从 3B 到 27B 的模型都提升 11--21 个百分点。
The gain is largest at the stages where a repair must restore the verifier-visible proof, showing that the diagnostic gives the model what a correct fix needs.
% 提升最大的阶段正是修复需要恢复 verifier 可见 proof 的地方，说明 diagnostic 给模型的正是正确修复需要的东西。

% 段7: 贡献
This paper makes the following contributions:
% 本文的贡献如下：
\begin{itemize}
\item An empirical study of 235 reproduced rejections showing the terminal error reports where verification stopped, leaving the developer to find where the proof was lost~(\S\ref{sec:study}).
% 一项对 235 个复现 rejection 的实证研究，表明 terminal error 报告的是 verification 停下的位置，开发者需自己找 proof 在哪丢的（见第 2 节）。
\item \sys, a technique that reconstructs where the required proof was established and lost from the verifier log~(\S\ref{sec:design}).
% \sys：一种从 verifier log 重建所需 proof 在哪建立、在哪丢失的技术（见第 3 节）。
\item \bench, a benchmark of 75 repair tasks with independent tests for evaluating LLM-based verifier repair~(\S\ref{sec:evaluation}).
% \bench：一个带独立测试的 75 修复任务 benchmark，用于评估基于 LLM 的 verifier 修复（见第 4 节）。
\end{itemize}
