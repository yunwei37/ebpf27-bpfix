\section{Introduction}
\label{sec:introduction}

eBPF lets developers run custom programs inside the Linux kernel for networking, security, and observability~\cite{gbadamosi2024ebpf, cilium}.
% eBPF 让开发者在内核里、网络/安全/可观测性的关键路径上运行自定义程序。
To load a program, a developer writes it in C or Rust, compiles it to eBPF bytecode, and submits it to a verifier. The verifier proves the program safe by tracking abstract values along every execution path~\cite{gershuni2019simple, linuxkernel-bpf-verifier}.
% 开发者用 C 或 Rust 写程序、编译成 BPF 字节码、提交给静态 verifier，verifier 沿每条执行路径跟踪抽象值来证明程序安全。

When the verifier rejects a program, the terminal error it returns identifies the rejection location, the operation where verification stopped, which can lie far from where the program lost the required proof.
% verifier 拒绝程序时，它返回的 terminal error 指出 rejection location，也就是 verification 停下来的那个 operation，而它可能离程序丢失所需 proof 的地方很远。
On rejection the verifier returns a log of bytecode and register states ending in this terminal error~\cite{linuxkernel-bpf-verifier}.
% 被拒时，verifier 返回一串字节码和寄存器状态的 log，末尾就是这行 terminal error。
In Figure~\ref{fig:overview}, a packet pointer that the program bounds-checks is rejected at a later load with \texttt{R5 invalid mem access `scalar'}, on a line the developer has no reason to suspect.
% 图 1 里，一个程序已经 bounds-check 过的 packet pointer，仍在后续一次 load 处被 \texttt{R5 invalid mem access 'scalar'} 拒绝，而出错的行开发者根本想不到有问题。
The distance between where the verifier reports the rejection and where the program lost the proof is the diagnostic gap.
% verifier 报告 rejection 的地方，和程序真正丢失 proof 的地方，二者之间的距离，就是 diagnostic gap。
Unable to close this gap from the terminal error alone, the developer resorts to repeated trial and error~\cite{deokar2024empirical}.
% 单凭 terminal error 没法弥合这个 gap，开发者只能反复试错。

Existing eBPF tools read the terminal error to recover the failing location, but none localizes where the required proof was lost~\cite{rizza2025design, bpfvv}.
% 现有 eBPF 工具靠读 terminal error 来恢复失败位置，但没有一个能定位 required proof 是在哪丢的。
PrettyVerifier maps the terminal error to a C source line and rewords it using the source and debug information~\cite{rizza2025design}.
% PrettyVerifier 把 terminal error 映到一个 C 源码行，并结合源码与调试信息把它重述。
The BPF Verifier Visualizer accumulates the verifier's per-instruction register states~\cite{bpfvv}.
% BPF Verifier Visualizer 把 verifier 逐指令的寄存器状态累积起来。
Even so, one terminal error comes from many root causes, and cannot show where the program lost the proof.
% 即便如此，一行 terminal error 来自许多 root cause，看不出程序在哪丢了这条 proof。

Recovering the lost proof is hard because the verifier exposes no structured artifact to localize against.
% 之所以难以恢复丢失的 proof，是因为 verifier 没有暴露任何可供定位的结构化制品。
Localizers in other settings walk a structure the checker builds, such as a source abstract syntax tree (AST) or typing constraints, but the eBPF verifier exposes none~\cite{pavlinovic2014finding, zhang2014toward}.
% 别的场景里，定位器会走一个 checker 本来就构建好的结构，例如源码 abstract syntax tree（AST）或类型约束，而 eBPF verifier 一个都不暴露。
The insight behind \sys is that the discarded proof survives in the log.
% \sys 的 insight 是：被丢弃的 proof 其实留在了 log 里。
The verifier builds a safety proof instruction by instruction and discards it at the rejection location, but under \texttt{log\_level=2} the verifier prints its abstract state after every instruction~\cite{linuxkernel-bpf-verifier}.
% verifier 逐指令构建一个 safety proof 并在 rejection location 处丢弃它，但在 \texttt{log\_level=2} 下它会在每条指令后打印它的抽象状态。
From the printed state, a tool reconstructs the lifecycle of the required proof, a structure that serves localization as a source AST does elsewhere.
% 从这些打印出的状态，工具就能重建 required proof 的生命周期，这个结构对定位的作用，正如源码 AST 之于别的定位器。

\sys is a userspace tool that localizes a rejection by reconstructing the proof lifecycle from the rejection log alone.
% \sys 是一个 userspace 工具，它只凭 rejection log 重建 proof 生命周期来定位 rejection。
From the log, \sys maps the terminal error to the required proof and replays it across the printed states, recording where the proof is established, lost, and needed.
% 从这份 log，\sys 把 terminal error 映成 required proof 并在打印出的状态上回放，记录 proof 在哪建立、在哪丢失、在哪被需要。
Figure~\ref{fig:overview} shows the resulting diagnostic, where \sys reports the established-to-lost transition as a proof-break witness across spans mapped back to the source.
% 图 1 给出这个诊断：\sys 把“建立到丢失”的转移点作为一个 proof-break witness，报告在若干映回源码的 span 上。

Replacing the raw verifier log with the \sys localization raises one-shot repair across the three downstream models we test, from 22 to 38 of 75 on the strongest, with each fix judged by an oracle independent of \sys.
% 把 raw verifier log 换成 \sys 的定位结果，在我们测试的三个下游模型上都提升了 one-shot 修复，最强模型从 75 例中的 22 提到 38，每个 fix 由一个独立于 \sys 的 oracle 判定。

This paper makes the following contributions:
% 本文的贡献如下：
\begin{itemize}
\item An empirical study on \bench showing the terminal error reports where verification stopped, leaving the proof loss to find~(\S\ref{sec:study}).
% 一项在 \bench 上的实证研究：表明 terminal error 报告的是 verification 停下来的位置，把 proof 的丢失点留给开发者去找（见第 2 节）。
\item \sys, a log-only, deterministic technique that localizes a rejection by reconstructing the required proof's lifecycle and reporting a proof-break witness~(\S\ref{sec:design}).
% \sys：一种只读 log、deterministic 的技术，靠重建 required proof 的生命周期、报告 proof-break witness 来定位 rejection（见第 3 节）。
\item \bench, 235 reproduced rejections with ground-truth fixes, and \testset, 75 repair tasks with \sys-independent oracles~(\S\ref{sec:study}, \S\ref{sec:evaluation}).
% \bench（235 个复现 rejection、带 ground-truth fix）与 \testset（75 个修复任务、带独立 oracle）两个数据集（见第 2、4 节）。
\end{itemize}
