\section{Related Work}\label{sec:related-work}

\noindent\textbf{eBPF verification and safety.}
Prior eBPF tooling decides whether a program is safe or works around the verifier, leaving rejection diagnosis to the developer~\cite{deokar2024empirical}.
% 既有 eBPF 工具要么判定程序是否安全、要么绕开 verifier，把 rejection 诊断留给开发者。
Offline re-provers such as PREVAIL establish safety over their own abstract domain~\cite{gershuni2019simple}, verifier audits check the soundness of its operators~\cite{vishwanathan2023verifying, vishwanathan2022sound, sun2024validating}, and other work replaces the verifier with safe Rust~\cite{jia2025rex}, synthesizes passing programs~\cite{zheng2024kgent}, or isolates programs at runtime~\cite{lu2024moat, zhang2024hive, lim2023unleashing, lim2024safebpf}.
% PREVAIL 在自己的抽象域上证明安全，verifier 审计检查它算子的可靠性，另一些工作用 safe Rust 替代 verifier、合成能过的程序、或在运行时隔离程序。
None replays the rejection log to reconstruct where the required proof was lost.
% 没有一个回放 rejection log 去重建 required proof 是在哪丢的。

\noindent\textbf{Error diagnosis and fault localization.}
Tools that diagnose rejections outside eBPF read facts the checker built: type-error diagnosis reasons over typing constraints~\cite{pavlinovic2014finding, zhang2014toward, seidel2017learning}, spectrum-based fault localization needs many test runs~\cite{wong2016survey, jones2005empirical}, program slicing needs the dependence graph~\cite{weiser1981program}, and unsat-core extraction needs a solver's search artifacts~\cite{cimatti2011computing}.
% eBPF 之外诊断 rejection 的工具读 checker 构建的 fact：type-error 诊断推 typing constraint，spectrum-based fault localization 需要大量测试运行，program slicing 需要依赖图，unsat-core 抽取需要 solver 的搜索制品。
For eBPF, Pretty Verifier uses regular expressions to map the terminal error to a source line and hint~\cite{rizza2025design}, and BPF Verifier Visualizer provides a frontend that renders per-instruction state interactively~\cite{bpfvv}, but neither localizes where the proof was lost.
% 在 eBPF 这边，Pretty Verifier 用正则把 terminal error 映到源码行，BPF Verifier Visualizer 提供一个前端交互式渲染逐指令状态，但都不定位 proof 在哪丢的。
\sys reconstructs the proof lifecycle from the verifier's own printed state, recovering register types, scalar ranges, and pointer provenance~\cite{linuxkernel-bpf-verifier}.
% \sys 从 verifier 自己打印的状态重建 proof 生命周期，恢复出寄存器类型、scalar 范围、指针 provenance。
