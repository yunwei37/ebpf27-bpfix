\section{Evaluation}
\label{sec:evaluation}

We evaluate \sys with three questions:
\begin{itemize}
  \item \textbf{RQ1.} Does a downstream model repair more programs from the \sys diagnostic than from the raw verifier log?
  \item \textbf{RQ2.} Does \sys produce a structured diagnostic across the rejections in \bench?
  \item \textbf{RQ3.} Can \sys identify the responsible layer when several layers share one verifier message?
\end{itemize}
RQ1 is the quantitative result, scored by an oracle independent of \sys. RQ2 and RQ3 characterize the diagnostic itself, reported as coverage and a case study.

\noindent\textbf{Datasets.} The evaluation uses two datasets, one per kind of question. RQ1 runs on \testset, 75 source-level repair tasks, 40 built around a verifier proof obligation and 35 minimized from real projects. An executable oracle scores each candidate by compiling it, loading it through the verifier, and running functional and source-semantics checks, none of which consults \sys. A model repairs each task from the buggy source plus either the raw log or the \sys diagnostic, the only difference between the prompts. RQ2 and RQ3 run on \bench (\S\ref{sec:study:corpus}), the 235 reproduced rejections, where the baselines are the raw verifier message and a dictionary that maps each terminal message to its most frequent layer.

\noindent\textbf{RQ1: Repairing the program.} Replacing the raw log with the \sys diagnostic raises executable repair. Under a fixed Qwen3.6 27B configuration, the model repairs 22 of the 75 cases from the raw verifier log at one attempt, and 38 from the \sys diagnostic (Figure~\ref{fig:bpfix-test}). A second attempt, a single failure-informed retry, raises the two to 30 and 44. The hosted GLM 5.2 shows the same one-attempt gain, 28 against 38, though the second attempt narrows the gap, 47 against 52. The small Qwen2.5 3B repairs none of the 75 from the raw log, 8 from the diagnostic in one shot, and 10 with one retry, which separates model capacity from diagnostic signal. Because the oracle checks the program itself, a passing repair is direct evidence that the diagnostic carried correct and usable repair signal that the raw log did not.

To see what the diagnostic fixes, Table~\ref{tab:failure-stage} records the stage at which each failed attempt stops. The \sys diagnostic mainly removes verifier-load and source-semantics failures: on the 27B one-attempt run, load failures fall from 19 to 10 and source-semantics failures from 22 to 16, while compile failures stay low. The diagnostic therefore helps the model both satisfy the verifier and preserve the program's behavior, the proof obligation the raw message leaves implicit. The model-call failures are a small-model effect: three raw prompts overflowed the 3B model's context window, so it returned no program, while the shorter diagnostic stayed within the window.

\noindent\textbf{RQ2: Diagnostic coverage.} \sys produces a structured diagnostic for every rejection in \bench, with no fallback, and the diagnostics span 21 error families. Each diagnostic adds a stable error id, the rejected source span, the safety proof the program must restore, and a concrete next action. Figure~\ref{fig:diag-example} shows a rejection whose terminal line reads only \texttt{only read from bpf\_array is supported}, where \sys recovers all four. These fields are what the downstream model receives beyond the raw log, which is why the diagnostic helps in RQ1. We report coverage and field presence: a field can be present and still point at the wrong place, so correctness comes from the independent oracle in RQ1.

\noindent\textbf{RQ3: Separating the responsible layer.} One verifier message can carry more than one responsible layer. The message \texttt{R invalid mem access 'scalar'} reaches \bench 28 times, as 24 source bugs and four compiler-lowering artifacts (Figure~\ref{fig:packet}). The four lowering cases are real upstream commits whose fix changes how the program is compiled (Table~\ref{tab:lowering}). A tool that reads only the message must label all 28 the same. The message dictionary labels every one a source bug. \sys reads the point in the trace where a packet or map pointer loses its provenance, and flags the four as lowering artifacts. The ground truth here is the upstream fix itself, which \sys did not produce. The verifier false positive layer is the hardest to ground, because calling a rejection a false positive rests on a human judgment that the program was correct, so we report that layer as a threat.

\noindent\textbf{Threats to validity.} \testset is a working suite built during development, so it is not a frozen heldout benchmark, and every number comes from one pinned kernel 6.15.11 with clang 18, whose verifier behavior is version-specific. The \sys diagnostic prompt is shorter than the raw log, so part of the repair gain could come from length; we leave the equal-budget trimmed-log control to future work. The \bench class labels were developed alongside \sys, so we do not report classification accuracy against them, which is why RQ2 scores field presence and RQ1 scores against the independent oracle. The labeled fix site lies within 13 instructions of the rejected line, so localization distance is not where \sys adds value. The lowering and false positive classes carry few cases, which we report as raw counts.
