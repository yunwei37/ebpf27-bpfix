\begin{abstract}
eBPF lets developers run custom programs inside the Linux kernel, but a static verifier must prove each program safe before it loads.
% eBPF 让开发者在 Linux 内核里运行自定义程序，但静态 verifier 必须在加载前证明每个程序安全。
When the verifier rejects a program, the unclear terminal error makes repair challenging: 47\% of rejections return only \texttt{EINVAL}, and one error string maps to as many as nine distinct root causes.
% 当 verifier 拒绝程序时，不清晰的 terminal error 使修复困难：47\% 的 rejection 只返回 \texttt{EINVAL}，一个 error 字符串最多对应 9 种不同的 root cause。
An empirical study of 235 reproduced rejections shows that 10 of 12 root causes are eBPF-specific, so repair requires domain knowledge beyond general programming.
% 一项对 235 个复现 rejection 的实证研究表明，12 类 root cause 中 10 类是 eBPF 特有的，因此修复需要超越通用编程的领域知识。
Existing tools help developers read the error, but none locates where the required proof was lost.
% 现有工具帮助开发者阅读 error，但没有一个能定位所需 proof 在哪丢失。
We present \sys, which reconstructs where the required proof was established and where it was lost from the verifier log.
% 我们提出 \sys：它从 verifier log 重建所需 proof 在哪建立、在哪丢失。
On a benchmark of 75 LLM repair tasks, current models achieve 0--37\% one-shot success with the raw log; replacing the log with the \sys localization improves repair by 11--21pp, suggesting that locating where the proof was lost is key to guiding repair.
% 在一个 75 个 LLM 修复任务的 benchmark 上，当前模型用 raw log 的 one-shot 成功率为 0--37\%；换成 \sys 的定位后提升 11--21 个百分点，说明定位 proof 在哪丢失是引导修复的关键。
\end{abstract}
